## 3. データモデル（論理設計）

### 3.1 エンティティ

**Hazard（ハザード）**

| 論理名 | 物理名 | 型 | 必須 | 説明 |
|--------|--------|-----|------|------|
| ID | id | String | ○ | CUID（自動生成） |
| 種別 | type | String | ○ | earthquake / tsunami / typhoon / flood / fire / volcanic / military / other |
| タイトル | title | String | ○ | ハザードの名称 |
| 詳細説明 | description | String | ○ | 詳細情報（空文字許容） |
| 緯度 | lat | Float | ○ | WGS84 緯度 |
| 経度 | lng | Float | ○ | WGS84 経度 |
| 場所 | prefecture | String | - | 都道府県・市区町村（任意） |
| 危険レベル | dangerLevel | Int | ○ | 1〜5 |
| ステータス | status | String | ○ | active / monitoring / resolved（デフォルト: active） |
| 発生日時 | occurredAt | DateTime | ○ | ハザード発生日時 |
| 解除日時 | resolvedAt | DateTime | - | ステータスが resolved の場合に設定 |
| 作成日時 | createdAt | DateTime | ○ | レコード作成日時（自動） |
| 更新日時 | updatedAt | DateTime | ○ | レコード最終更新日時（自動・楽観的ロックに使用） |

### 3.2 危険レベルと色の対応

| レベル | 色 | ラベル | 意味 |
|--------|-----|--------|------|
| 1 | 緑 (#22c55e) | 低危険 | 軽微な注意 |
| 2 | 黄 (#eab308) | 注意 | 注意が必要 |
| 3 | 橙 (#f97316) | 警戒 | 警戒が必要 |
| 4 | 赤 (#ef4444) | 危険 | 直ちに対処 |
| 5 | 暗赤 (#991b1b) | 最高危険 | 最大級の対応 |

## 4. API設計（概要）

### 4.1 エンドポイント一覧

すべての `/api/hazards` エンドポイントはミドルウェアで認証チェックを行い、未認証の場合は `401 Unauthorized` を返す。

| メソッド | パス | 概要 |
|---------|------|------|
| GET/POST | /api/auth/[...nextauth] | NextAuth 認証（ログイン・ログアウト・セッション） |
| GET | /api/hazards | ハザード一覧取得（フィルタ対応） |
| POST | /api/hazards | ハザード新規作成 |
| GET | /api/hazards/:id | ハザード詳細取得 |
| PUT | /api/hazards/:id | ハザード更新（楽観的ロック付き） |
| DELETE | /api/hazards/:id | ハザード削除 |

### 4.2 楽観的ロック方式

```
クライアント                    サーバー
    |                              |
    |-- PUT { ...data, updatedAt ->|
    |                              |-- Zod バリデーション
    |                              |-- DBの updatedAt と比較
    |                              |
    |                 一致した場合 --|-- UPDATE 実行 --> 200 OK
    |                              |
    |             不一致の場合 ----|-- 409 Conflict (現在のレコードを返却)
    |                              |
    |      ユーザーが「強制保存」選択|
    |-- PUT { ...data } (updatedAt なし) -->|
    |                              |-- 競合チェックなしで UPDATE
    |                              |--> 200 OK
```

### 4.3 入力バリデーション方針

Zod スキーマで全 API リクエストを検証する。バリデーション失敗時は `400 Bad Request` とエラー詳細を返す。

| フィールド | バリデーションルール |
|-----------|-------------------|
| type | 許可された8種別のみ（enum） |
| title | 1〜200文字 |
| dangerLevel | 整数 1〜5 |
| lat / lng | lat: -90〜90、lng: -180〜180 |
| occurredAt | ISO 8601 形式の日時文字列 |
| status | active / monitoring / resolved のみ |

## 5. 認証設計

### 5.1 認証方式

NextAuth.js（v4）の CredentialsProvider を使用する。

- **セッション方式:** JWT（データベース不要）
- **有効期間:** 24時間
- **認証情報:** 環境変数 `ADMIN_USERNAME` / `ADMIN_PASSWORD` と照合

### 5.2 アクセス制御フロー

```
リクエスト
    |
    |-- /api/auth/*  ──────────────────> NextAuth 処理（認証不要）
    |-- /login       ──────────────────> ログインページ表示（認証不要）
    |
    |-- その他すべて
    |     |
    |     |-- JWT トークン検証（middleware）
    |     |     |
    |     |     |-- 有効 ──────────────> 処理継続
    |     |     |
    |     |     |-- 無効・未ログイン
    |     |           |
    |     |           |-- /api/* ──────> 401 Unauthorized
    |     |           |-- ページ ──────> /login へリダイレクト
```

### 5.3 必要な環境変数

| 変数名 | 説明 | 例 |
|--------|------|----|
| `NEXTAUTH_SECRET` | JWT 署名シークレット | `openssl rand -base64 32` で生成 |
| `NEXTAUTH_URL` | アプリケーションの公開URL | `https://your-app.vercel.app` |
| `ADMIN_USERNAME` | 管理者ユーザー名 | `admin` |
| `ADMIN_PASSWORD` | 管理者パスワード | 任意の強力なパスワード |
| `DATABASE_URL` | PostgreSQL 接続文字列 | `postgresql://user:pass@host/db?sslmode=require` |

## 6. コンポーネント構成

### 6.1 コンポーネント階層

```
RootLayout (layout.tsx)
└── Providers (Providers.tsx)  ← SessionProvider ラッパー
    ├── LoginPage (login/page.tsx)  ← 未認証時
    └── HomePage (page.tsx)  ← 状態管理の中枢（認証済み時）
        ├── HazardHistory    ← 左サイドバー（一覧・フィルタ・ログアウト）
        ├── JapanMap (dynamic, SSR=false)  ← 地図エリア
        │   └── HazardMarker × N  ← 各ハザードのピン
        ├── HazardDetailPanel  ← 右パネル（詳細・編集）
        │   └── DangerBadge
        └── HazardForm (modal)  ← 新規登録フォーム
```

### 6.2 状態管理方針

- **グローバル状態:** `page.tsx` の useState/useRef で一元管理
- **ローカル状態:** 各コンポーネントのフォーム状態・UIトグル等
- **ポーリング管理:** `useRef` で編集中フラグを管理し、ポーリング時の上書きを防止
- **セッション管理:** `SessionProvider` 経由で NextAuth セッションをグローバルに提供

## 7. 外部サービス連携

### 7.1 OpenStreetMap（地図タイル）

- タイルURL: `https://{s}.tile.openstreetmap.org/{z}/{x}/{y}.png`
- クライアント（ブラウザ）から直接リクエスト
- 利用条件: タイル利用規約への準拠・著作権表示

### 7.2 Nominatim（逆ジオコーディング）

- エンドポイント: `https://nominatim.openstreetmap.org/reverse`
- パラメータ: `format=json`, `lat`, `lon`, `accept-language=ja`
- 地図クリック時に1回リクエスト（AbortController でキャンセル可能）
- 利用制限: 1リクエスト/秒（本システムは人間操作ベースのため問題なし）
- レスポンスから住所組み立て: `state` + `city/town/village` + `city_district/suburb`
  - `state` が取れない場合は `display_name` から都道府県名を正規表現で抽出

## 8. 実行環境・起動手順

### 8.1 起動（開発）

```bash
docker compose up --build  # 初回（postgres イメージのダウンロードを含む）
docker compose up          # 2回目以降
```

### 8.2 コンテナ起動シーケンス

```
1. postgres コンテナ起動（ヘルスチェック通過まで待機）
2. app コンテナ起動
   2-1. npm install (postinstall: prisma generate)
   2-2. npx prisma generate（Prismaクライアント生成）
   2-3. npx prisma db push（スキーマを PostgreSQL に反映）
   2-4. npm run dev（Next.js 開発サーバー起動）
```

### 8.3 データ永続化

- **開発:** Docker ボリューム `postgres_data`（コンテナ削除後もデータ保持）
- **本番:** Neon / Supabase 等の外部 PostgreSQL サービス

### 8.4 Vercel デプロイ手順

1. Neon 等で PostgreSQL データベースを作成し `DATABASE_URL` を取得
2. Vercel プロジェクトに以下の環境変数を設定（`.env.example` 参照）
   - `DATABASE_URL`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL`, `ADMIN_USERNAME`, `ADMIN_PASSWORD`
3. GitHub リポジトリと Vercel プロジェクトを連携してデプロイ
