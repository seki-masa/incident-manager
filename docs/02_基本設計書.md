## 3. データモデル（論理設計）

### 3.1 エンティティ

**Hazard（ハザード）**

| 論理名 | 物理名 | 型 | 必須 | 説明 |
|--------|--------|-----|------|------|
| ID | id | String | ○ | CUID（自動生成） |
| 種別 | type | String | ○ | earthquake / tsunami / typhoon / flood / fire / volcanic / military / other |
| タイトル | title | String | ○ | ハザードの名称 |
| 詳細説明 | description | String | ○ | 詳細情報（空文字許容） |
| 緯度 | lat | Float | ○ | WGS84 緯度 |
| 経度 | lng | Float | ○ | WGS84 経度 |
| 場所 | prefecture | String | - | 都道府県・市区町村（任意） |
| 危険レベル | dangerLevel | Int | ○ | 1〜5 |
| ステータス | status | String | ○ | active / monitoring / resolved（デフォルト: active） |
| 発生日時 | occurredAt | DateTime | ○ | ハザード発生日時 |
| 解除日時 | resolvedAt | DateTime | - | ステータスが resolved の場合に設定 |
| 作成日時 | createdAt | DateTime | ○ | レコード作成日時（自動） |
| 更新日時 | updatedAt | DateTime | ○ | レコード最終更新日時（自動・楽観的ロックに使用） |

### 3.2 危険レベルと色の対応

| レベル | 色 | ラベル | 意味 |
|--------|-----|--------|------|
| 1 | 緑 (#22c55e) | 低危険 | 軽微な注意 |
| 2 | 黄 (#eab308) | 注意 | 注意が必要 |
| 3 | 橙 (#f97316) | 警戒 | 警戒が必要 |
| 4 | 赤 (#ef4444) | 危険 | 直ちに対処 |
| 5 | 暗赤 (#991b1b) | 最高危険 | 最大級の対応 |

## 4. API設計（概要）

### 4.1 エンドポイント一覧

| メソッド | パス | 概要 |
|---------|------|------|
| GET | /api/hazards | ハザード一覧取得（フィルタ対応） |
| POST | /api/hazards | ハザード新規作成 |
| GET | /api/hazards/:id | ハザード詳細取得 |
| PUT | /api/hazards/:id | ハザード更新（楽観的ロック付き） |
| DELETE | /api/hazards/:id | ハザード削除 |

### 4.2 楽観的ロック方式

```
クライアント                    サーバー
    |                              |
    |-- PUT { ...data, updatedAt ->|
    |                              |-- DBの updatedAt と比較
    |                              |
    |                 一致した場合 --|-- UPDATE 実行 --> 200 OK
    |                              |
    |             不一致の場合 ----|-- 409 Conflict (現在のレコードを返却)
    |                              |
    |      ユーザーが「強制保存」選択|
    |-- PUT { ...data } (updatedAt なし) ->|
    |                              |-- 競合チェックなしで UPDATE
    |                              |--> 200 OK
```

## 5. コンポーネント構成

### 5.1 コンポーネント階層

```
HomePage (page.tsx)  ← 状態管理の中枢
├── HazardHistory    ← 左サイドバー（一覧・フィルタ）
├── JapanMap (dynamic, SSR=false)  ← 地図エリア
│   └── HazardMarker × N  ← 各ハザードのピン
├── HazardDetailPanel  ← 右パネル（詳細・編集）
│   └── DangerBadge
└── HazardForm (modal)  ← 新規登録フォーム
```

### 5.2 状態管理方針

- **グローバル状態:** `page.tsx` の useState/useRef で一元管理
- **ローカル状態:** 各コンポーネントのフォーム状態・UIトグル等
- **ポーリング管理:** `useRef` で編集中フラグを管理し、ポーリング時の上書きを防止

## 6. 外部サービス連携

### 6.1 OpenStreetMap（地図タイル）

- タイルURL: `https://{s}.tile.openstreetmap.org/{z}/{x}/{y}.png`
- クライアント（ブラウザ）から直接リクエスト
- 利用条件: タイル利用規約への準拠・著作権表示

### 6.2 Nominatim（逆ジオコーディング）

- エンドポイント: `https://nominatim.openstreetmap.org/reverse`
- パラメータ: `format=json`, `lat`, `lon`, `accept-language=ja`
- 地図クリック時に1回リクエスト（AbortController でキャンセル可能）
- 利用制限: 1リクエスト/秒（本システムは人間操作ベースのため問題なし）
- レスポンスから住所組み立て: `state` + `city/town/village` + `city_district/suburb`
  - `state` が取れない場合は `display_name` から都道府県名を正規表現で抽出

## 7. 実行環境・起動手順

### 7.1 起動

```bash
docker-compose up --build  # 初回
docker-compose up          # 2回目以降
```

### 7.2 コンテナ起動シーケンス

```
1. npm install
   └── postinstall: prisma generate
2. npx prisma generate（Prismaクライアント生成）
3. npx prisma db push（スキーマをDBに反映）
4. npm run dev（Next.js 開発サーバー起動）
```

### 7.3 データ永続化

- SQLite DB ファイルパス（ホスト）: `./prisma/prisma/dev.db`
- Docker ボリュームマウント: `./prisma:/app/prisma`
- コンテナ削除後もデータは保持される
